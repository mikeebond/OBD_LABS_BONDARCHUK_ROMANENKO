Звіт з нормалізації бази даних (Лабораторна робота 5)
Проєкт: Вебзастосунок для ведення особистого бюджету.

1. Аналіз початкової (ненормалізованої) схеми
Початкова схема бази даних була спроєктована у попередніх лабораторних роботах. Вона вже частково нормалізована, але має кілька недоліків.

Функціональні залежності (ФЗ) початкової схеми:

Users: user_id -> name_surname, email, mobile_number, password_hash, registration_date

Accounts: account_id -> user_id, account_name, balance, currency

Transactions: transaction_id -> account_id, category_id, amount, transaction_date, description

Transaction_Tags: (transaction_id, tag_id) -> (немає інших атрибутів)

Найвища нормальна форма початкової схеми:
Схема частково знаходиться у 1NF, але має порушення строгих правил 1NF та потенційні проблеми з 3NF.

2. Перехід до Першої нормальної форми (1NF)
Правило 1NF: Усі атрибути є атомарними (без повторюваних груп). Якщо таблиця має стовпець, який може містити кілька значень або складну структуру, це порушує 1NF.

Порушення: У таблиці Users атрибут name_surname містить відразу два факти: ім'я та прізвище. Це не атомарне значення.

Рішення: Розділяємо цей стовпець на два окремих атрибути: first_name та last_name.

Команда для зміни (ALTER TABLE):

SQL
ALTER TABLE users DROP COLUMN name_surname;
ALTER TABLE users ADD COLUMN first_name VARCHAR(50) NOT NULL;
ALTER TABLE users ADD COLUMN last_name VARCHAR(50) NOT NULL;

3. Перехід до Другої нормальної форми (2NF)
Правило 2NF: Таблиця повинна знаходитися в 1NF, і всі її неключові атрибути мають повністю залежати від усього первинного ключа. Не повинно бути часткових залежностей (коли атрибут залежить лише від частини складеного ключа).

Аналіз: Більшість таблиць у нашій базі даних (users, accounts, transactions, categories, budgets, goals) мають простий (одинарний) штучний первинний ключ (наприклад, user_id, account_id). Оскільки ключ не складений, часткової залежності в них бути не може за визначенням.

Аналіз таблиці зі зв'язком багато-до-багатьох: Єдина таблиця зі складеним первинним ключем — це transaction_tags (ключ складається з transaction_id та tag_id). Проте ця таблиця не містить жодних інших неключових атрибутів. Тому часткових залежностей там також не існує.

Висновок: Наша схема бази даних автоматично задовольняє вимоги 2NF після перевірки на 1NF. Жодних структурних змін на цьому етапі робити не потрібно.

4. Перехід до Третьої нормальної форми (3NF)
Правило 3NF: Таблиця повинна знаходитися в 2NF, і в ній не має бути транзитивних залежностей. Тобто жоден неключовий атрибут не повинен залежати від іншого неключового атрибута. Усі факти мають описувати лише первинний ключ.

Порушення: У таблиці Accounts ми зберігали валюту рахунку у текстовому полі currency (наприклад, 'UAH', 'USD'). Якби ми захотіли зберігати ще й символ валюти (₴, $) або її повну назву, ці дані залежали б від коду валюти, а не від самого рахунку (account_id). Зберігання таких даних безпосередньо в таблиці Accounts призвело б до масового дублювання та аномалій оновлення (наприклад, при зміні символу валюти довелося б оновлювати тисячі рядків).

Рішення: Виносимо інформацію про валюти в окрему таблицю-довідник currencies. У самій таблиці accounts залишаємо лише зовнішній ключ currency_code, який посилається на новий довідник.
